# Plan 12-01: Database Schema & Supabase Infrastructure

**Goal:** Create Supabase database schema for user profiles, follows, and plants with Row Level Security (RLS) policies and Storage buckets.

## Tasks

### Task 1: Create database migration SQL files

Create two migration files in `supabase/migrations/`:

**001_initial_schema.sql** - Core tables:
- `profiles` table with columns:
  - `id UUID PRIMARY KEY REFERENCES auth.users(id)`
  - `display_name TEXT NOT NULL CHECK(char_length(display_name) <= 50)`
  - `bio TEXT CHECK(char_length(bio) <= 500)`
  - `avatar_url TEXT`
  - `created_at TIMESTAMPTZ DEFAULT now()`
  - `updated_at TIMESTAMPTZ DEFAULT now()`
- `follows` table with columns:
  - `follower_id UUID REFERENCES profiles(id)`
  - `following_id UUID REFERENCES profiles(id)`
  - `created_at TIMESTAMPTZ DEFAULT now()`
  - `PRIMARY KEY (follower_id, following_id)`
- `plants` table (see migrationService.ts lines 199-216 for full schema)
- `watering_history` table (see migrationService.ts lines 242-249 for full schema)

**002_profiles_follows_tables.sql** - RLS policies and triggers:
- Enable RLS on all tables
- Create policies for profiles:
  - SELECT: Authenticated users can read all profiles
  - UPDATE: Users can update only their own profile (`auth.uid() = id`)
  - INSERT: No direct inserts (handled by trigger)
- Create policies for follows:
  - SELECT: Users can view follows where they are follower or following
  - INSERT: Users can create follows where they are the follower
  - DELETE: Users can delete follows where they are the follower
- Create policies for plants and watering_history:
  - Full CRUD for user's own data (`user_id = auth.uid()`)
- Create Postgres trigger function `handle_new_user()` to auto-create profile on signup
  - Default display_name from email: `split_part(NEW.email, '@', 1)`
  - Trigger fires on `auth.users INSERT`
- Create updated_at trigger function for profiles table

### Task 2: Create Storage buckets configuration

Create `lib/supabase/storage.ts` with bucket setup functions:

**Export functions:**
- `initializeStorageBuckets()` - Creates buckets if they don't exist:
  - `avatars` bucket: Public bucket, user-isolated folders (`{userId}/avatar.jpg`)
  - `plant-photos` bucket: Public bucket, user-isolated folders (`{userId}/{plantId}/`)
  - Both buckets have RLS policies for authenticated read/write

**RLS policies for storage:**
- `avatars` bucket:
  - SELECT: Public read (avatars are visible to all authenticated users)
  - INSERT: Users can upload only to their own folder (`bucket_id = 'avatars' AND (storage.foldername(name))[1] = auth.uid()::text`)
  - UPDATE: Same as INSERT
  - DELETE: Same as INSERT

- `plant-photos` bucket:
  - SELECT: Public read
  - INSERT/UPDATE/DELETE: Users can manage only their own folder

### Task 3: Create Supabase profile queries layer

Create `lib/supabase/profiles.ts` with typed query functions:

**Type definitions:**
```typescript
interface Profile {
  id: string;
  display_name: string;
  bio: string | null;
  avatar_url: string | null;
  created_at: string;
  updated_at: string;
}

interface ProfileWithStats extends Profile {
  plants_identified: number;
  followers_count: number;
  following_count: number;
}

interface Follow {
  follower_id: string;
  following_id: string;
  created_at: string;
}
```

**Export functions:**
- `getProfile(userId: string): Promise<Profile | null>` - Fetch single profile
- `getProfileWithStats(userId: string): Promise<ProfileWithStats | null>` - Fetch profile with counts
  - Use join syntax: `plants(count)`, `followers:follows!follows_following_id_fkey(count)`, `following:follows!follows_follower_id_fkey(count)`
  - Return shape matches ProfileWithStats interface
- `updateProfile(userId: string, updates: Partial<Pick<Profile, 'display_name' | 'bio' | 'avatar_url'>>): Promise<void>`
- `followUser(followerId: string, followingId: string): Promise<Follow>`
- `unfollowUser(followerId: string, followingId: string): Promise<void>`
- `isFollowing(followerId: string, followingId: string): Promise<boolean>`

**Error handling:**
- Throw errors with user-friendly messages
- Return null for profile not found (not an error)

### Task 4: Create TypeScript types for profiles

```typescript
export interface Profile {
  id: string;

  setIsEditing: (isEditing: boolean) => void;
  setLoading: (loading: boolean) => void;
  setError: (error: string | null) => void;
  clearProfile: () => void;
}
```

Export these types from `types/index.ts` for app-wide use.

### Task 5: Create profile state management store

Create `stores/profileStore.ts` following authStore pattern:

**Store structure:**
```typescript
import { create } from 'zustand';
import type { ProfileState } from '@/types/profile';

export const useProfileStore = create<ProfileState>((set, get) => ({
  currentProfile: null,
  isEditing: false,
  isLoading: false,
  error: null,

  setCurrentProfile: (profile) => set({ currentProfile: profile }),
  setIsEditing: (isEditing) => set({ isEditing }),
  setLoading: (isLoading) => set({ isLoading }),
  setError: (error) => set({ error }),
  clearProfile: () => set({ currentProfile: null, error: null }),
}));
```

**No persist middleware** - Profile data fetched fresh from Supabase on app launch.
